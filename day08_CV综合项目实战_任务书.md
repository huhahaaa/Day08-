## 一、项目基本信息

| 项目 | 要求 |
|---|---|
| 项目名称 | CV 综合项目实战：智能数字媒体内容审核与视频精彩片段提取 |
| 开发形式 | 五、六人小组协作开发 |

```text

    -> 输出文件和分析报告
```

## 二、项目背景

数字媒体内容团队每天会处理大量游戏录屏、场景演示、宣传素材和用户投稿内容。仅依靠人工逐段观看视频，会产生以下问题：

- 长视频中很难快速定位角色、战斗、道具和重要场景；
- 内容审核结果缺乏统一标准，无法追溯具体证据；
- 任务处理过程没有任务编号和状态记录；
- 视频、关键帧、检测结果和测试资料容易分散；
- 交付时缺少统一的文件命名、报告和验收依据。

本项目要求围绕真实的内容生产流程，开发一个轻量级 CV 工作台。系统不要求达到商业产品的完整规模，但必须体现企业项目中的基本工程能力：需求拆解、模块分工、接口约定、异常处理、测试验证和结果交付。

## 四、方向选择

每组只能选择一个方向作为本日主项目。两个方向共用环境、任务状态、文件存储、前端工作台和测试规范，但核心业务不同。

### 方向 A：智能数字媒体内容审核系统

#### 4.1 业务目标

面向游戏宣传素材、数字媒体视频和图片素材，建立一套“机器初筛 + 人工复核 + 结果留档”的内容审核流程。

#### 4.2 必须实现的功能

1. 支持上传图片或视频素材；
2. 使用 YOLO 对图片或视频采样帧进行目标检测；
3. 记录检测类别、置信度和边界框坐标；
4. 根据目标类别、置信度和审核规则进行内容筛选；
5. 输出以下三种审核结果：

```text
通过
待复核
不通过
```

6. 保存至少一张或多张审核证据帧；
7. 页面展示原始素材、检测结果、审核结论和证据图片；
8. 支持人工修改审核结论，并将修改结果写回 JSON；
9. 输出可重新读取的审核报告；
10. 对不支持格式、空文件、模型不存在和任务失败给出明确提示。

#### 4.3 建议审核规则

小组需要在 `PRD.md` 中明确规则，不允许只写“根据模型判断”。可采用以下基础规则并根据项目场景调整：

```text
发现预先配置的高风险类别，或高风险类别置信度 >= 0.60
    -> 不通过

检测结果置信度处于 0.35 到 0.60 之间，或多个采样帧结果不稳定
    -> 待复核

未发现规则目标，且处理过程正常
    -> 通过
```

审核规则至少要配置以下内容：

```json
{
  "risk_classes": ["person"],
  "reject_confidence": 0.60,
  "review_confidence": 0.35,
  "min_evidence_frames": 1
}
```

#### 4.4 方向 A 加分功能

- 支持页面修改审核阈值；
- 支持审核规则配置化；
- 支持批量处理多张图片；
- 生成通过、待复核、不通过数量统计；
- 导出 CSV 或 ZIP 审核包；
- 支持审核备注和负责人字段；
- 支持审核结果版本记录。

### 方向 B：智能视频精彩片段提取系统

#### 4.5 业务目标

面向游戏录屏、游戏场景演示和宣传素材，利用目标检测、画面变化和运动强度，自动找出值得人工复核的关键帧和候选片段。

#### 4.6 必须实现的功能

1. 支持上传游戏录屏或场景演示视频；
2. 使用 OpenCV 按时间间隔读取视频帧；
3. 使用 YOLO 检测每个采样帧中的目标；
4. 计算目标数量、画面变化和运动强度；
5. 生成综合精彩分数；
6. 按分数筛选关键帧，并避免同一时间段重复选择；
7. 自动生成一段推荐视频；
8. 支持人工对关键帧执行保留或忽略；
9. 支持关键帧排序、标签和备注；
10. 支持调整片段开始时间和结束时间；
11. 使用 FFmpeg 输出粗剪视频；
12. 至少支持横屏 16:9 和另外一种输出规格；
13. 输出关键帧联系表、分析报告和视频文件；
14. 任务完成后能够从历史记录重新打开并继续审核。

#### 4.7 推荐评分规则

推荐评分必须可解释，至少包含目标密度和画面变化两个指标。推荐使用以下基础公式：

```text
highlight_score =
    object_score * 0.45
  + scene_change_score * 0.35
  + motion_score * 0.20
```

指标说明：

| 指标 | 含义 |
|---|---|
| `object_score` | 当前采样帧检测到的目标数量归一化分数 |
| `scene_change_score` | 当前帧与相邻采样帧的画面差异 |
| `motion_score` | 画面运动强度或连续帧变化程度 |
| `highlight_score` | 三项指标加权后的最终推荐分数 |

#### 4.8 推荐片段边界要求

推荐片段必须满足：

```text
0 <= start < end <= video_duration
```

推荐策略必须处理以下场景：

- 最高分关键帧靠近视频开头；
- 最高分关键帧靠近视频结尾；
- 用户设置的目标时长大于原视频时长；
- 视频没有音频轨道；
- 关键帧数量不足；
- 连续多个高分帧来自同一个镜头。

当源视频足够长时，推荐片段应尽量达到用户设置的目标时长，不能因为最高分关键帧靠近结尾而意外只输出几秒。

#### 4.9 方向 B 加分功能

- 多个候选片段合并；
- 片段拖动或按钮调整顺序；
- 横屏、竖屏和方形三种发布规格；
- 自动生成 AI 封面 Prompt；
- 生成视频、关键帧和 JSON 报告的交付 ZIP；
- 片段标签统计；
- 原音频保留检测；
- 推荐策略参数可调并保存到任务报告。

## 五、共享系统设计要求

### 5.1 任务状态

所有小组的系统都必须实现以下任务状态：

```text
created -> queued -> running -> completed
                              └-> failed
```

页面需要至少显示：

- 任务编号；
- 项目名称；
- 素材名称；
- 当前状态；
- 创建时间；
- 开始时间和结束时间；
- 处理失败时的错误信息。

上传接口不能让浏览器一直等待完整的模型推理。创建任务后，应先返回任务编号，再通过轮询或其他方式查询处理状态。

### 5.2 任务 JSON

每个任务目录必须保存一个 `job.json`，至少包含以下字段：

```json
{
  "job_id": "20260717_101530_a1b2c3d4",
  "project_name": "星港遗迹内容审核",
  "asset_name": "opening_scene.mp4",
  "status": "completed",
  "created_at": "2026-07-17T10:15:30",
  "started_at": "2026-07-17T10:15:31",
  "completed_at": "2026-07-17T10:16:08",
  "settings": {},
  "result_file": "analysis_report.json",
  "error": null
}
```

失败任务也必须保留 `job.json`，不能因为失败就只在控制台输出异常。

### 5.3 目录结构

建议使用以下目录结构：

```text
team_project/
├─ app.py
├─ requirements.txt
├─ models/
│  └─ yolo11n.pt
├─ assets/
├─ outputs/
│  └─ job_id/
│     ├─ input/
│     ├─ keyframes/
│     ├─ result/
│     ├─ job.json
│     └─ analysis_report.json
├─ static/
├─ templates/
├─ tests/
└─ docs/
   ├─ PRD.md
   ├─ SYSTEM_DESIGN.md
   ├─ API.md
   ├─ TEST_REPORT.md
   └─ BUG_RECORD.md
```

### 5.4 最低 API 要求

#### 公共接口

```text
GET    /api/health
POST   /api/jobs
GET    /api/jobs
GET    /api/jobs/<job_id>
DELETE /api/jobs/<job_id>
```

#### 方向 A 接口

```text
POST   /api/jobs/<job_id>/analyze
PATCH  /api/jobs/<job_id>/review
GET    /api/jobs/<job_id>/report
```

#### 方向 B 接口

```text
POST   /api/jobs/<job_id>/analyze
PATCH  /api/jobs/<job_id>/review
POST   /api/jobs/<job_id>/rough-cut
GET    /api/jobs/<job_id>/report
```

接口统一要求：

- 成功响应包含 `ok: true`；
- 失败响应包含 `ok: false` 和可读的 `error`；
- 使用合适的 HTTP 状态码；
- 上传接口先返回任务编号，不阻塞等待完整推理；
- 处理结果可以通过任务编号重新查询；
- 删除接口只能删除合法任务目录；
- 正在 `queued` 或 `running` 的任务不能直接删除；
- 接口参数错误不能导致服务进程退出。

### 5.5 技术边界

本日项目必须使用：

- Python：业务逻辑和服务端程序；
- Flask：网页和 HTTP API；
- HTML/CSS/JavaScript：桌面端工作台；
- OpenCV：读取视频、采样帧和画面分析；
- Ultralytics YOLO：目标检测；
- FFmpeg：视频剪辑、编码、合成和音频处理；
- JSON 和本地文件夹：任务状态和结果持久化。

自定义模型，训练 YOLO。
