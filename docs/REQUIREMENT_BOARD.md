# Day08 智能数字媒体内容审核系统需求看板

## 1. 看板用途

本看板用于学校单日实训中的小组分工、进度同步和验收追踪。需求范围来自 `docs/PRD.md`，编号保持一致。当前代码仓库仅具备 Flask、CV、任务管理和前端骨架，因此开发需求的初始状态统一按“未开始”记录，成员应在完成真实代码和验证后更新状态。

## 2. 状态和更新规则

| 状态 | 含义 | 进入条件 |
|---|---|---|
| 未开始 | 尚未产生可运行产出 | 需求已明确且已指定主责角色 |
| 进行中 | 已开始开发，但还不能联调 | 已有本地代码或文档变更 |
| 待联调 | 单模块自测通过，等待上下游接口 | 模块产出、调用说明和自测证据完整 |
| 待验收 | 联调通过，等待产品和测试确认 | 对应验收场景可以执行 |
| 已完成 | 已满足完成定义并留存证据 | 产品与测试共同确认，代码已经提交 |
| 受阻 | 因依赖、环境或接口问题无法继续 | 已记录阻塞原因、责任人和解除条件 |

更新规则：

1. 主责角色负责更新状态，协作角色负责提供依赖产出。
2. 需求进入“待联调”前必须完成模块自测，不能仅以代码已编写作为依据。
3. 需求进入“已完成”前必须关联 Git commit、测试记录或页面截图。
4. 发现阻塞时立即记录，不用“进行中”掩盖无法推进的问题。
5. P0 需求优先于 P1；P0 最小闭环未打通时不得投入新的扩展功能。

## 4. 需求任务卡

### REQ-001 上传素材并创建任务

- **业务价值：** 建立所有后续处理的统一入口和任务标识。
- **主责/协作：** 后端主责；前端完成上传表单；测试准备合法与非法素材。
- **接口：** `POST /api/jobs`。
- **开发子任务：**
  1. 后端校验项目名称、文件存在性、扩展名和文件大小；
  2. 生成 job_id 和任务目录，保存输入文件与 job.json；
  3. 前端提交 `multipart/form-data`，展示上传中和表单错误；
  4. 测试合法图片、合法视频、空文件和错误格式。
- **完成定义：** 接口返回 HTTP 201 和 job_id；输入文件与 job.json 可读取；上传接口不等待模型推理；正常与异常证据已保存。
- **验收依据：** PRD REQ-001、验收清单 AC-001。

### REQ-002 异步处理与状态查询

- **业务价值：** 避免浏览器长时间阻塞，并让处理过程可观察。
- **主责/协作：** 后端主责；前端实现轮询；测试验证状态和失败分支。
- **接口：** `POST /api/jobs/<job_id>/analyze`、`GET /api/jobs/<job_id>`。
- **开发子任务：**
  1. 实现合法状态转换和时间戳更新；
  2. 使用后台线程或任务执行器运行分析；
  3. 前端轮询 created、queued、running 任务，并在结束状态停止；
  4. 测试正常完成、分析异常和非法状态转换。
- **完成定义：** 可观测到 created、queued、running、completed/failed 的合法变化；失败原因写入 job.json；前端状态自动刷新。
- **验收依据：** PRD REQ-002、验收清单 AC-002。

### REQ-003 图片和视频 YOLO 检测

- **业务价值：** 用真实 CV 推理替代纯人工逐帧查看。
- **主责/协作：** CV 主责；后端传递输入路径；测试提供图片、视频和损坏文件。
- **接口：** 由分析任务调用 `CVEngine`，结果通过任务详情和报告间接提供。
- **开发子任务：**
  1. 校验并加载 `models/yolo11n.pt`；
  2. 实现图片推理和统一检测结构；
  3. 使用 FFmpeg/ffprobe 获取视频时长、分辨率、编码和音频轨道信息；
  4. 使用 OpenCV 按规则间隔读取视频帧并推理；
  5. 校验类别、置信度、边界框、帧号和时间点。
- **完成定义：** 合法图片和视频产生真实模型结果；视频报告包含媒体信息；坐标与置信度合法；模型缺失、FFmpeg 探测失败和视频解码失败进入 failed。
- **验收依据：** PRD REQ-003、验收清单 AC-003。

### REQ-004 配置化审核规则与结论

- **业务价值：** 让机器结论有统一且可说明的判断依据。
- **主责/协作：** CV 主责；产品确认规则；后端保存规则快照；测试验证边界值。
- **接口：** 分析任务内部规则评估，结果写入 `analysis_report.json`。
- **开发子任务：**
  1. 读取风险类、警告类和阈值配置；
  2. 按 reject、review、pass 优先级聚合检测；
  3. 保存触发类别、置信度、规则和原因；
  4. 测试 0.35、0.60 和无命中场景。
- **完成定义：** 同一输入与规则重复评估得到相同结论；三个结论均有测试；报告可解释触发原因。
- **验收依据：** PRD REQ-004、验收清单 AC-004。

### REQ-005 证据帧与审核报告留档

- **业务价值：** 为机器判断和人工复核保留可追溯证据。
- **主责/协作：** CV 生成证据；后端管理路径和报告接口；测试核对文件引用。
- **接口：** `GET /api/jobs/<job_id>/report`。
- **开发子任务：**
  1. 为规则命中结果绘制类别、置信度和检测框；
  2. 图片或视频证据写入任务 keyframes 目录；
  3. 生成 UTF-8 analysis_report.json；
  4. 校验报告引用文件存在且能通过页面访问。
- **完成定义：** 风险任务至少保存配置要求数量的证据；报告可重读；失败任务仍保留错误 job.json。
- **验收依据：** PRD REQ-005、验收清单 AC-005。

### REQ-006 审核结果可视化

- **业务价值：** 让审核员在单一工作台理解机器结论并定位证据。
- **主责/协作：** 前端主责；后端提供详情和文件访问；CV确认检测字段含义；产品确认信息层级。
- **接口：** `GET /api/jobs/<job_id>`、`GET /api/jobs/<job_id>/report`。
- **开发子任务：**
  1. 实现任务基本信息和状态区域；
  2. 展示机器结论、原因、检测明细和证据图片；
  3. 统一通过、待复核、不通过的文字和颜色；
  4. 完成加载、空结果、完成和失败视图。
- **完成定义：** 页面数据与 JSON 一致；不同状态均有可见反馈；证据可查看；接口错误不只写入控制台。
- **验收依据：** PRD REQ-006、验收清单 AC-006。

### REQ-007 人工复核并写回结果

- **业务价值：** 形成“机器初筛 + 人工最终确认”的业务闭环。
- **主责/协作：** 后端主责写回；前端完成复核表单；产品确认字段；测试验证持久化。
- **接口：** `PATCH /api/jobs/<job_id>/review`。
- **开发子任务：**
  1. 校验任务完成状态和人工结论枚举；
  2. 保存人工结论、备注、复核人和复核时间；
  3. 保留机器结论和原证据；
  4. 前端提交后重新读取报告并展示最终结论。
- **完成定义：** 服务重启后人工复核仍可读取；非法结论和非完成任务被拒绝；机器原始判断不丢失。
- **验收依据：** PRD REQ-007、验收清单 AC-007。

### REQ-008 历史任务重开与删除

- **业务价值：** 支持内容负责人追踪历史结果并安全清理任务。
- **主责/协作：** 后端主责；前端完成列表和操作；测试验证服务重启与路径安全。
- **接口：** `GET /api/jobs`、`GET /api/jobs/<job_id>`、`DELETE /api/jobs/<job_id>`。
- **开发子任务：**
  1. 从 outputs 扫描并按时间倒序返回任务；
  2. 从 JSON 重建历史详情和最终结论；
  3. 禁止删除 created、queued、running 任务；
  4. 校验 job_id，保证删除目标不越出 outputs。
- **完成定义：** 重启服务后历史可打开；合法已结束任务可删除；非法、未知或运行中任务不会删除任何目录。
- **验收依据：** PRD REQ-008、验收清单 AC-008。

### REQ-009 异常输入和失败提示

- **业务价值：** 让用户知道失败原因，并避免单任务错误影响整个服务。
- **主责/协作：** 后端统一错误响应；CV抛出可解释异常；前端展示；测试覆盖异常矩阵。
- **接口：** 所有 API。
- **开发子任务：**
  1. 统一 `ok: false` 和非空 error 响应；
  2. 区分参数错误、资源不存在、状态冲突和服务异常状态码；
  3. 捕获模型、OpenCV 和文件异常并写入失败任务；
  4. 页面在表单或任务详情附近显示错误。
- **完成定义：** 不支持格式、空文件、模型缺失、任务失败和非法删除均有稳定提示；健康检查和其他任务不受影响。
- **验收依据：** PRD REQ-009、验收清单 AC-009。

### REQ-010 页面维护审核规则

- **业务价值：** 在不修改代码的情况下适配不同内容审核标准。
- **主责/协作：** 前端主责交互；后端负责持久化和校验；CV消费规则；产品确认约束；测试验证新旧任务隔离。
- **接口：** 规则读写接口由后端在 API 文档中补充，任务创建继续通过 `settings` 保存规则快照。
- **开发子任务：**
  1. 完成风险类别、警告类别、阈值、证据数量和采样间隔表单；
  2. 实现前后端一致的范围校验；
  3. 支持保存配置和恢复默认规则；
  4. 验证规则变更只影响新任务。
- **完成定义：** 合法配置可保存并用于新任务；非法输入被明确拒绝；旧任务报告中的规则快照不变。
- **验收依据：** PRD REQ-010、验收清单 AC-010。

## 8. 每日同步与阻塞记录

每次联调同步只回答以下四项：

1. 上一阶段完成了什么，可提供哪条 commit 或测试证据；
2. 下一阶段要把哪项需求推进到什么状态；
3. 当前阻塞是什么，由谁在什么时间前解除；
4. 是否影响 P0 最小闭环和 16:20 最终验收。

阻塞项使用下表记录，解除后保留记录用于复盘：

| 记录字段 | 填写要求 |
|---|---|
| 关联需求 | 使用 REQ 编号 |
| 发现时间 | 精确到分钟 |
| 问题现象 | 写明可观察到的结果 |
| 影响范围 | 指出受影响的接口、模块或验收项 |
| 责任角色 | 指定一个主责角色 |
| 解除条件 | 写成可以验证的结果 |
| 最终证据 | 关联 commit、测试记录或截图 |

## 9. 看板完成判定

本项目只有在 REQ-001 至 REQ-009 全部进入“已完成”、P0 验收项有证据、真实 YOLO 分析闭环可演示时，才能认定为完成。REQ-010 是本组承诺的加分功能，应在 P0 稳定后验收；如果 P0 尚有受阻项，REQ-010 不得占用核心联调时间。
