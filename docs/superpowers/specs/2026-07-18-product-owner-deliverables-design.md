# 产品负责人交付物设计规格

## 1. 背景

项目沿用正式名称“Day08 智能数字媒体内容审核系统”，选择任务书方向 A，面向学校单日实训和小组 GitHub 协作场景。产品负责人的交付目标是形成可直接指导开发、测试、验收和现场演示的产品文档，不扩展到代码实现。

## 2. 产品范围

本次产品定义覆盖方向 A 的完整必做闭环，并纳入“审核规则配置化”一项加分功能：

1. 上传图片或视频并创建任务；
2. 后台执行 YOLO 检测并记录类别、置信度和边界框；
3. 根据配置化规则输出通过、待复核或不通过；
4. 保存证据帧和可重新读取的 JSON 报告；
5. 展示任务状态、素材、检测结果和审核结论；
6. 支持人工修改结论、填写备注和记录复核人；
7. 支持历史任务查询与合规删除；
8. 明确处理格式错误、空文件、模型缺失和任务失败。

批量处理、CSV/ZIP 导出、版本历史等其他加分功能不进入本次承诺范围。

## 3. 交付物结构

| 模块 | 文件 | 作用 |
|---|---|---|
| 产品需求 | `docs/PRD.md` | 定义用户、场景、范围、流程、规则、状态和非功能需求 |
| 需求看板 | `docs/REQUIREMENT_BOARD.md` | 将需求拆成可分工、可跟踪、可验收的任务项 |
| 验收清单 | `docs/ACCEPTANCE_CHECKLIST.md` | 提供按功能、异常、接口、页面和交付物检查的验收依据 |
| 演示流程 | `demo/demo_script.md` | 提供 8 分钟现场演示顺序、讲解要点和异常预案 |

## 4. 文档一致性约束

- 三种审核结论统一使用“通过 / 待复核 / 不通过”，接口值对应 `pass / review / reject`。
- 任务状态统一使用 `created -> queued -> running -> completed`，异常进入 `failed`。
- 基础规则沿用仓库 `default_rules.json`，风险类别和阈值可在页面配置并保存到任务设置。
- API 路径以任务书和 `docs/API.md` 中声明的方向 A 接口为准。
- 验收项必须能够追溯到 PRD 需求编号，需求看板也使用相同编号。
- 未提供真实成员姓名时，负责人以产品、后端、CV、前端、测试角色标记。

## 5. Git 协作策略

- 所有内容只提交到 `黄林` 分支，不改动 `master`。
- 四份正式产出分别形成独立 commit，并在提交后推送至 `origin/黄林`。
- 提交信息使用中文 Conventional Commits 风格，体现单一模块范围。
- 任务书当前为未跟踪文件，不纳入任何提交。
- 每次提交前检查暂存区，确保不夹带其他成员或无关文件。

## 6. 完成标准

产品负责人交付完成需同时满足：

1. 四份正式产出均存在且无未完成标记或空白占位；
2. PRD、看板、验收清单之间的需求编号一致；
3. 演示流程覆盖正常闭环、人工复核、历史重开和异常提示；
4. 每个正式模块都有独立 Git 提交并已推送至 `origin/黄林`；
5. 文档内容与任务书方向 A、现有 API 骨架和规则配置保持一致。
