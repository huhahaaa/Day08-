## 2. 项目背景

游戏宣传素材、场景演示视频和用户投稿内容需要在发布前完成内容审核。完全依赖人工逐段观看会造成审核耗时长、判断标准不统一、问题证据难追溯和处理记录分散。

本项目建设一个轻量级内容审核工作台，以“机器初筛 + 人工复核 + 结果留档”为核心：系统使用 YOLO 检测图片或视频采样帧中的目标，根据配置化规则生成初审结论，再由审核人员查看证据并决定是否修改结论。每次处理均以任务编号组织输入、状态、证据和报告，保证结果可查询、可解释、可复核。

## 3. 产品目标与成功标准

### 3.1 产品目标

1. 将素材上传、模型分析、规则判断、人工复核和报告留档整合为一条完整流程。
2. 用统一阈值和风险类别降低不同审核人员之间的判断差异。
3. 用检测框、置信度和证据帧说明机器结论，避免只给出不可解释的结果。
4. 用任务状态和 JSON 报告保存处理过程，便于历史查询和项目验收。

### 3.2 成功标准

- 上传一份合法图片或视频后，系统立即返回任务编号，页面无需等待完整推理。
- 任务能够从创建状态进入处理中，并最终进入完成或失败状态。
- 完成任务至少展示一项真实 CV 检测结果或明确的无风险结果，并保存可重新读取的 JSON 报告。
- 人工复核后，结论、备注和复核人能够写回报告并在重新打开任务时保留。
- 格式错误、空文件、模型缺失和处理失败均有可读提示，失败任务仍保留错误记录。

## 5. 产品范围

### 5.1 V1.0 范围

- 单个图片或视频素材上传；
- 创建异步审核任务并查询状态；
- 使用 YOLO 检测图片或视频采样帧；
- 保存检测类别、置信度和边界框；
- 使用配置化风险类别和阈值生成审核结论；
- 保存至少一张有效证据帧；
- 展示素材信息、检测结果、机器结论和证据图片；
- 人工修改结论并记录备注、复核人和复核时间；
- 保存、查询和重新读取任务报告；
- 删除已完成或已失败的合法任务；
- 在页面维护审核规则，并把任务采用的规则保存到任务设置。

### 5.2 不在 V1.0 范围

- 多素材批量上传和并行批处理；
- CSV、ZIP 审核包导出；
- 审核结论的多版本历史和回滚；
- 用户登录、权限和组织管理；
- 云存储、数据库和线上部署；
- 在线训练或自动更新 YOLO 模型。

## 6. 核心业务流程

```text
进入任务列表
  -> 新建任务并填写项目名称
  -> 选择图片或视频、确认审核规则
  -> 上传素材并获得 job_id
  -> created -> queued -> running
  -> YOLO 检测 + 规则评估 + 证据帧保存
  -> completed 或 failed
  -> 查看检测框、置信度、机器结论和证据
  -> 人工确认或修改结论，填写备注和复核人
  -> 写回 JSON 报告
  -> 从历史任务重新打开并继续查看
```

## 7. 审核规则

### 7.1 默认配置

默认值与 `default_rules.json` 保持一致：

```json
{
  "risk_classes": ["knife", "scissors", "baseball bat"],
  "warning_classes": ["bottle", "wine glass", "cell phone"],
  "reject_confidence": 0.60,
  "review_confidence": 0.35,
  "min_evidence_frames": 1,
  "video_sample_interval_sec": 1.0
}
```

### 7.2 判定优先级

规则按“不通过 > 待复核 > 通过”的优先级执行：

1. 任一风险类别置信度 `>= 0.60`，结论为“不通过（reject）”。
2. 未触发不通过，但风险类别置信度满足 `0.35 <= confidence < 0.60`，结论为“待复核（review）”。
3. 任一警告类别置信度 `>= 0.35`，结论为“待复核（review）”。
4. 视频多个采样帧的规则命中不稳定，或有效证据帧少于配置数量，结论不得自动通过，应进入待复核。
5. 处理正常且没有触发以上规则，结论为“通过（pass）”。
6. 置信度 `< 0.35` 的结果可以保留在原始检测列表中，但不触发审核结论。

每个任务必须保存当次实际使用的规则快照。修改全局规则只影响之后创建的任务，不回写既有任务。

## 8. 功能需求

### REQ-001 上传素材并创建任务

**用户故事：** 作为内容审核员，我希望上传一份图片或视频并立即获得任务编号，以便后续查询处理结果。

**前置条件：** 服务健康检查通过；用户位于任务列表页。

**处理规则：**

- 支持常见图片格式 `.jpg`、`.jpeg`、`.png` 和视频格式 `.mp4`、`.avi`、`.mov`、`.mkv`。
- 项目名称必填并去除首尾空格；素材文件必选且不能为空。
- 服务端生成不可由用户指定的 `job_id`，文件保存到该任务的 `input` 目录。
- 创建接口只负责保存文件和创建任务，不等待完整模型分析。

**验收标准：**

- 合法素材上传成功时返回 HTTP 201、`ok: true`、`job_id` 和初始任务信息。
- `outputs/<job_id>/job.json` 与输入文件存在，素材原始名称记录在 `asset_name`。
- 不支持格式、空文件或缺少参数时返回 4xx、`ok: false` 和可读错误信息。

### REQ-002 异步处理与状态查询

**用户故事：** 作为内容负责人，我希望随时查看任务进度，而不是让上传页面一直等待。

**前置条件：** 合法任务已创建。

**处理规则：**

- 状态按 `created -> queued -> running -> completed` 流转；处理异常时由 `queued` 或 `running` 进入 `failed`。
- `created_at` 在创建时写入；`started_at` 在进入 `running` 时写入；`completed_at` 在进入 `completed` 或 `failed` 时写入。
- 前端在任务未结束时轮询详情接口，并对不同状态显示明确文案。

**验收标准：**

- 上传接口在后台分析完成前返回，页面能看到 job_id。
- 状态变化能够通过任务详情接口查询，页面无需刷新即可更新。
- 非法状态转换被拒绝，不破坏已有 `job.json`。

### REQ-003 图片和视频 YOLO 检测

**用户故事：** 作为内容审核员，我希望系统识别素材中的目标，以减少逐帧人工观看。

**前置条件：** 任务处于 queued；YOLO 模型加载成功；输入文件可读取。

**处理规则：**

- 图片执行一次 YOLO 推理。
- 视频先使用 FFmpeg/ffprobe 读取时长、分辨率、编码和音频轨道等媒体信息，再使用 OpenCV 按 `video_sample_interval_sec` 采样并逐帧推理。
- 每条检测结果记录 `class_name`、`confidence` 和 `[x1, y1, x2, y2]` 边界框；视频结果额外记录 `frame_idx` 和 `time_sec`。
- 坐标不得超出对应图像边界，置信度范围为 0 至 1。

**验收标准：**

- 使用真实 YOLO 模型处理合法素材，结果结构符合约定。
- 视频报告包含 FFmpeg/ffprobe 获取的媒体信息，并能够产生带时间位置的采样记录，采样间隔来自任务规则。
- 模型缺失或素材无法解码时任务进入 failed，并保存错误原因。

### REQ-004 配置化审核规则与结论

**用户故事：** 作为内容负责人，我希望审核结论由明确规则产生，以便解释和复现。

**前置条件：** 检测完成；任务包含有效规则快照。

**处理规则：** 按第 7 节的判定优先级聚合全部检测结果，输出 `pass`、`review` 或 `reject`，并记录触发类别、置信度、原因和关联证据。

**验收标准：**

- 风险类别置信度恰好为 0.60 时判定 reject。
- 风险类别置信度恰好为 0.35 时判定 review。
- 未触发风险或警告规则且处理正常时判定 pass。
- 报告能够说明命中的规则，而不是只保存结论。

### REQ-005 证据帧与审核报告留档

**用户故事：** 作为内容审核员，我希望看到并保存支持机器结论的证据，以便后续追溯。

**前置条件：** 模型分析已产生检测或无风险结论。

**处理规则：**

- 对触发风险或警告规则的图片、视频帧绘制检测框并保存到 `keyframes` 目录。
- 证据帧数量不少于任务规则的 `min_evidence_frames`；确实无检测结果时，报告明确记录零命中，不伪造证据。
- `analysis_report.json` 至少包含任务信息、规则快照、检测结果、机器结论、结论原因、证据文件和人工复核字段。
- `job.json.result_file` 指向报告文件。

**验收标准：**

- 完成任务的报告是合法 UTF-8 JSON，可通过报告接口重新读取。
- 报告引用的证据文件真实存在且可由页面访问。
- 失败任务仍保留 `job.json`，其中包含状态、完成时间和错误信息。

### REQ-006 审核结果可视化

**用户故事：** 作为内容审核员，我希望在一个页面内理解素材、检测结果和机器结论。

**前置条件：** 任务详情或报告可读取。

**处理规则：**

- 审核工作台展示任务编号、项目名称、素材名称、状态和时间信息。
- 完成状态展示机器结论、原因、检测类别、置信度、边界框和证据图片。
- 页面区分加载中、空结果、完成和失败状态；不能把接口错误留在浏览器控制台而不提示用户。

**验收标准：**

- 点击历史任务可进入审核工作台并看到与 JSON 一致的数据。
- 三种结论使用清晰且一致的文字和颜色表达。
- 无检测结果和任务失败时页面均有专门提示。

### REQ-007 人工复核并写回结果

**用户故事：** 作为内容审核员，我希望确认或修改机器结论，并留下复核依据。

**前置条件：** 任务已 completed，报告存在。

**处理规则：**

- 人工结论只能为 pass、review 或 reject。
- 提交时记录 `manual_conclusion`、`review_note`、`reviewer` 和 `reviewed_at`。
- 人工结论不覆盖 `machine_conclusion`，两者同时保留；页面最终结论优先展示人工结论。

**验收标准：**

- 合法复核提交返回 `ok: true`，重新请求报告时仍能读到相同结果。
- 非法结论值或未完成任务的复核请求返回 4xx 和明确提示。
- 机器结论及其证据在人工修改后仍然保留。

### REQ-008 历史任务重开与删除

**用户故事：** 作为内容负责人，我希望查询历史任务、重新打开结果，并清理已经结束的任务。

**前置条件：** outputs 目录包含任务记录。

**处理规则：**

- 列表按创建时间倒序显示任务编号、项目名称、素材名称、状态、创建时间和最终结论。
- 重新打开任务时从本地 JSON 读取，不依赖浏览器内存中的旧数据。
- 只允许删除 completed 或 failed 任务；created、queued、running 任务不得删除。
- `job_id` 必须通过格式和目录归属校验，删除目标必须位于 outputs 下的对应任务目录。

**验收标准：**

- 服务重启后，历史任务仍可列出、打开并读取报告。
- 删除合法已结束任务后，该任务目录和列表项消失。
- 删除运行中任务、未知任务或包含路径字符的 job_id 时返回 4xx/404，且其他目录不受影响。

### REQ-009 异常输入和失败提示

**用户故事：** 作为系统使用者，我希望系统清楚说明失败原因，以便采取正确措施。

**前置条件：** 用户提交输入或后台处理遇到异常。

**处理规则：**

- API 失败响应统一包含 `ok: false` 和非空 `error`。
- 不支持格式、空文件和参数错误属于客户端错误，使用合适的 4xx 状态码。
- 模型不存在、解码失败或推理异常使任务进入 failed；服务进程不得退出。
- 前端在表单附近或任务详情中展示错误，不仅使用弹窗或控制台日志。

**验收标准：**

- 四类必测异常均能稳定复现并得到可读错误。
- 后台任务失败后，任务列表可看到 failed，详情可看到错误原因。
- 单个任务失败不影响健康检查和其他任务查询。

### REQ-010 页面维护审核规则

**用户故事：** 作为系统维护人员，我希望在页面调整审核类别和阈值，以适应不同审核场景。

**前置条件：** 用户打开规则配置弹窗。

**处理规则：**

- 页面可编辑风险类别、警告类别、拒绝阈值、复核阈值、最小证据帧数和视频采样间隔。
- 阈值满足 `0 <= review_confidence < reject_confidence <= 1`；最小证据帧数为不小于 1 的整数；采样间隔大于 0。
- 保存前进行前端和后端双重校验；创建任务时把规则副本写入 `settings`。
- V1.0 规则配置保存在本地配置文件，不要求多用户隔离。

**验收标准：**

- 合法配置保存后，新建任务使用新规则，已有任务继续使用原规则快照。
- 非法阈值或空类别项不会被保存，并显示具体校验原因。
- 页面能够恢复默认规则。

## 10. 数据与任务状态

每个任务至少保存以下字段：

```json
{
  "job_id": "20260718_101530_a1b2c3d4",
  "project_name": "Day08 智能数字媒体内容审核系统",
  "asset_name": "opening_scene.mp4",
  "status": "completed",
  "created_at": "2026-07-18T10:15:30",
  "started_at": "2026-07-18T10:15:31",
  "completed_at": "2026-07-18T10:16:08",
  "settings": {},
  "result_file": "analysis_report.json",
  "error": null
}
```

状态转换约束：

| 当前状态 | 允许进入 | 禁止操作 |
|---|---|---|
| created | queued | 删除、人工复核 |
| queued | running、failed | 删除、人工复核 |
| running | completed、failed | 删除、人工复核 |
| completed | 无后续处理状态 | 再次触发分析 |
| failed | 无后续处理状态 | 人工复核 |

## 11. 页面信息架构

### 11.1 任务列表页

- 顶部：系统名称、健康状态、新建任务、规则配置；
- 状态概览：全部、处理中、已完成、失败数量；
- 任务表格：任务编号、项目名称、素材、状态、创建时间、结论、操作；
- 空状态：说明如何创建第一个任务；
- 新建任务弹窗：项目名称、素材、规则摘要、提交和错误信息。

### 11.2 审核工作台

- 任务信息区：编号、素材、状态、创建/开始/结束时间；
- 素材与证据区：原始预览、证据图片、检测框；
- 检测明细区：类别、置信度、坐标、视频时间点；
- 审核结论区：机器结论、原因、人工结论、备注、复核人；
- 操作区：返回列表、刷新、提交复核、读取报告。

## 12. 非功能需求

| 编号 | 类别 | 要求 |
|---|---|---|
| NFR-001 | 响应性 | 上传创建接口不得阻塞等待完整模型推理，前端使用轮询更新状态 |
| NFR-002 | 可解释性 | 每个风险结论必须关联检测类别、置信度、规则原因和证据 |
| NFR-003 | 持久性 | 服务重启后仍可通过本地 JSON 重新读取历史任务 |
| NFR-004 | 安全性 | 文件名和 job_id 必须校验，任何文件操作不得越出任务目录 |
| NFR-005 | 健壮性 | 单个任务异常不得导致 Flask 服务退出，失败任务必须留档 |
| NFR-006 | 一致性 | API 响应、页面文案、报告字段使用统一状态和结论枚举 |
| NFR-007 | 易用性 | 页面必须提供加载、空结果、完成、失败和表单错误状态 |
