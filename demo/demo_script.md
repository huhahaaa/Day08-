# Day08 智能数字媒体内容审核系统 8 分钟演示流程

## 1. 演示目标

在 8 分钟内向指导教师完整展示“上传素材 → 创建异步任务 → YOLO 检测 → 规则判断 → 证据展示 → 人工复核 → 报告留档 → 历史重开 → 异常提示”的方向 A 核心闭环，并说明小组角色分工和工程化处理。

演示不是单纯展示页面效果，必须让观众看见：

1. 真实任务编号和状态变化；
2. 真实 YOLO 检测类别、置信度和边界框；
3. 可解释的规则阈值、机器结论和证据帧；
4. 人工复核结果写回 JSON；
5. 历史任务可重新读取，异常情况有明确记录。

## 2. 演示角色

| 角色 | 演示职责 |
|---|---|
| 产品负责人 | 开场、业务目标、流程串联、人工复核、范围取舍和总结 |
| 后端工程师 | 上传接口、job_id、状态机、历史查询和本地 JSON |
| CV 工程师 | FFmpeg 媒体探测、OpenCV 采样、YOLO 检测和规则解释 |

| 测试工程师 | 异常输入、失败留档、测试与 Bug 证据 |

如果现场由一人主讲，产品负责人按本脚本连续操作，其他成员只在问答时补充技术细节。

## 5. 逐段操作与讲解词

### 00:00-00:45 项目介绍（45 秒）

**操作者：** 产品负责人。

**页面操作：** 停留在任务列表首页，鼠标指向系统名称、状态概览和“新建审核任务”按钮。

**讲解词：**

> 我们的项目是“Day08 智能数字媒体内容审核系统”，面向游戏宣传素材、场景演示视频和用户投稿内容。传统审核依赖人工逐段观看，效率低、标准不统一，结论也缺少证据。我们的方案是机器初筛加人工复核：YOLO 负责发现目标，配置化规则输出通过、待复核或不通过，审核员查看证据后给出最终结论。所有输入、状态、证据和报告都按任务编号保存，便于追溯。

**必须展示：** 项目名称、方向 A、三种结论、桌面端工作台。

**对应需求：** REQ-001 至 REQ-010。

### 00:45-01:30 架构与分工（45 秒）

**操作者：** 产品负责人打开架构图，后端工程师补充一句数据流。

**页面操作：** 切换到系统架构图，按浏览器、Flask、任务管理、CV 引擎、本地 JSON 的顺序指示。

**讲解词：**

> 前端通过 Flask API 上传和查询任务；JobManager 管理 created、queued、running、completed 或 failed 状态，并把过程写入 job.json；视频先由 FFmpeg/ffprobe 读取媒体信息，再由 OpenCV 按间隔采样，YOLO 完成检测，规则引擎生成结论和证据；最终 analysis_report.json 可被页面和历史任务重新读取。团队按产品、后端、CV、前端和测试分工，每个模块都有独立 Git 提交。

**必须展示：** 异步任务、本地持久化、FFmpeg/OpenCV/YOLO 的职责边界。

**对应需求：** REQ-002、REQ-003、REQ-005。

### 01:30-02:30 创建任务（60 秒）

**操作者：** 前端工程师操作，后端工程师讲解。

**页面操作：**

1. 返回任务列表并点击“新建审核任务”；
2. 输入项目名称“课堂演示内容审核”；
3. 选择 `demo_risk.jpg`；
4. 展开规则摘要，指出风险类、0.60 拒绝阈值和 0.35 复核阈值；
5. 点击创建，立即指出返回的 job_id 和初始状态。

**讲解词：**

> 创建接口只保存素材和任务信息，先返回 job_id，不让浏览器等待完整推理。每个任务都保存当次使用的规则快照，因此以后修改全局规则不会改变历史结论。规则页面允许维护类别、阈值、证据数量和视频采样间隔，这是我们选择实现的加分功能。

**预期画面：** 弹窗关闭，新任务出现在列表中，能看到 job_id 和 created/queued 状态。

**对应需求：** REQ-001、REQ-002、REQ-010。

### 02:30-04:00 机器审核（90 秒）

**操作者：** CV 工程师主讲，后端工程师指出状态变化。

**页面操作：** 打开新任务详情或在列表观察轮询；依次指出 queued、running 和 completed。若图片处理过快看不到中间状态，打开浏览器网络记录或预先保存的状态截图说明。

**讲解词：**

> 后台任务进入 running 后加载真实 YOLO 模型。图片直接推理；如果是视频，会先用 ffprobe 获取时长、分辨率、编码和音频轨道，再由 OpenCV 按配置间隔采样。每条检测记录类别、置信度和边界框，视频还记录帧号和时间点。规则按不通过、待复核、通过的优先级评估：风险类别达到 0.60 直接不通过，0.35 到 0.60 进入待复核，没有触发规则且处理正常才通过。

**必须展示：**

- 任务状态真实变化；
- 模型名称和就绪状态；
- 至少一条真实检测数据；
- 0.35 和 0.60 两个阈值的边界含义。

**对应需求：** REQ-002、REQ-003、REQ-004。

### 04:00-05:15 结果解释（75 秒）

**操作者：** 前端工程师展示页面，CV 工程师解释结果。

**页面操作：**

1. 指向机器审核结论和原因；
2. 打开检测明细，指出 class_name、confidence 和 bbox；
3. 点击或放大证据图片，指出检测框；
4. 切换到一个无规则命中的 pass 历史任务，说明空结果状态。

**讲解词：**

> 系统不会只显示一个结论。审核员可以看到触发类别、实际置信度、采用的阈值、检测坐标和证据帧。当前结论之所以是这里显示的结果，是因为这条检测命中了对应规则。报告保存的是规则快照和原始机器结论，所以同一个结果可以被重新解释。对于没有命中规则的任务，系统明确显示零命中和通过原因，不伪造证据。

**预期画面：** 页面数据与 analysis_report.json 一致，证据文件可打开，三种结论的文字和颜色清楚。

**对应需求：** REQ-004、REQ-005、REQ-006。

### 05:15-06:15 人工复核（60 秒）

**操作者：** 产品负责人操作，前端工程师补充写回结果。

**页面操作：**

1. 在 completed 任务中选择一个与机器结论不同的人工结论；
2. 备注填写“课堂演示：证据可接受，转为待复核”；
3. 复核人填写“黄林”；
4. 提交并刷新任务；
5. 指向人工结论、机器结论和复核时间。

**讲解词：**

> 产品定位不是让模型替代人，而是让模型先筛选。人工复核保存 manual_conclusion、备注、复核人和时间，不覆盖 machine_conclusion 和原始证据。最终页面优先展示人工结论，但审计时仍能看到机器当时为什么这样判断。

**预期画面：** 提交成功；人工结论更新；机器结论仍然存在；重新读取后复核信息不丢失。

**对应需求：** REQ-007。

### 06:15-07:00 历史任务与报告（45 秒）

**操作者：** 后端工程师。

**页面操作：**

1. 返回任务列表并打开一个演示前准备的历史任务；
2. 展示其报告接口或本地 analysis_report.json；
3. 指出 outputs 下 input、keyframes、result、job.json 的组织方式；
4. 指出删除按钮只对 completed/failed 任务可用，但现场不必删除备用结果。

**讲解词：**

> 任务历史不是保存在浏览器内存里，而是从本地 JSON 重新读取。服务重启后仍能恢复任务、规则、机器结论和人工复核。删除接口只允许清理已完成或已失败任务，并校验 job_id 与目标路径，运行中的任务不能删除。

**预期画面：** 历史数据与报告一致，文件引用真实存在。

**对应需求：** REQ-005、REQ-008。

### 07:00-07:30 异常处理（30 秒）

**操作者：** 测试工程师。

**页面操作：** 在新建任务弹窗选择 `invalid.txt` 并提交，然后快速打开预先准备的 failed 任务。

**讲解词：**

> 错误格式和空文件在创建阶段返回 4xx 和可读提示；模型缺失、媒体探测失败或推理异常会让任务进入 failed，并把错误写入 job.json。单个任务失败不会让 Flask 服务退出。测试报告覆盖至少 5 条正常、5 条异常和两个真实 Bug 的修复回归。

**预期画面：** 表单附近显示格式错误；failed 任务显示 error；服务仍可访问。

**对应需求：** REQ-009。

### 07:30-08:00 总结（30 秒）

**操作者：** 产品负责人。

**页面操作：** 回到包含多种状态和结论的任务列表。

**讲解词：**

> 我们完成的核心价值不是单次模型识别，而是一条可运行、可解释、可复核、可追溯的内容审核闭环。P0 覆盖上传、异步状态、真实检测、规则判断、证据报告、人工复核、历史任务和异常处理；在此基础上增加了规则配置化。项目通过模块化代码、统一 API、测试记录和独立 Git 提交体现小组协作过程。

**对应需求：** REQ-001 至 REQ-010。

## 6. 现场故障预案

### 6.1 模型加载失败

**现象：** `/api/health` 显示 model_ready 为 false，或分析任务直接失败。

**处理：**

1. 不临时编造检测数据，也不修改报告伪装成功；
2. 展示健康检查和 failed 任务中的模型路径错误，说明失败留档机制；
3. 打开演示前生成的 completed 真实任务，继续讲解检测结果和证据；
4. 演示后检查模型文件路径、Ultralytics 依赖和当前 Conda 环境。

### 6.2 视频处理时间过长

**现象：** 新视频在演示时间内保持 queued 或 running。

**处理：**

1. 用实时任务展示异步状态和页面轮询；
2. 切换到预先完成的同一视频任务展示 ffprobe 信息、采样帧、检测和报告；
3. 明确说明备用任务是预先真实运行的结果，不声称现场任务已经完成。

### 6.3 浏览器无法访问页面

**现象：** 本地页面打不开或静态资源加载失败。

**处理：**

1. 检查启动终端和 `/api/health`；
2. 使用 Flask 测试客户端或 API 工具展示任务接口和报告；
3. 用准备好的页面截图说明预期交互；
4. 不在现场重新安装依赖或大范围修改代码。

### 6.4 新任务处理失败

**现象：** 现场上传成功，但分析进入 failed。

**处理：**

1. 先展示页面错误和 job.json，说明错误可追溯；
2. 切换到 completed 历史任务完成结果、复核和报告演示；
3. 保留现场失败任务，作为演示后的 Bug 分析材料。

## 7. 常见提问与回答

### Q1：为什么上传后不直接等待分析结果？

模型推理和视频采样耗时不稳定。异步任务可以先返回 job_id，让浏览器查询状态，也能在失败时保留完整任务记录，符合任务书的工程要求。

### Q2：三种审核结论是如何得到的？

系统先聚合 YOLO 检测，再按规则优先级判断。风险类别置信度不低于 0.60 为不通过；风险类别在 0.35 至 0.60 之间或警告类别不低于 0.35 为待复核；处理正常且没有触发规则才通过。报告同时保存命中项、阈值和原因。

### Q3：人工修改后还能追溯机器结果吗？

可以。人工字段与 machine_conclusion 分开保存，人工修改不会覆盖模型结论、原因或证据帧，服务重启后也能从报告重新读取。

### Q4：删除任务如何避免误删其他目录？

后端只接受符合白名单格式的 job_id，解析目标绝对路径后必须确认仍位于 outputs 下，并且只允许删除 completed 或 failed 任务。

### Q5：为什么方向 A 还使用 FFmpeg？

FFmpeg/ffprobe 负责读取视频时长、分辨率、编码和音频轨道等媒体信息；OpenCV 负责按时间间隔采样；YOLO 负责目标检测。三者边界明确，也满足任务书的必选技术栈要求。

### Q6：为什么只选择规则配置化这一项加分功能？

单日实训优先保证 P0 闭环。规则配置化直接增强核心审核流程，又与现有 default_rules.json 和规则弹窗骨架一致；批量、导出和版本历史会增加联调风险，因此不纳入 V1.0 承诺。

### Q7：如何证明这是团队真实协作，而不是一次性拼装？

需求看板明确了角色、依赖和完成定义；各模块有独立 commit；API、测试、Bug 和截图可以与 Git 历史相互对应，每位成员都能说明自己的产出和联调关系。
